# SEO Architecture

This document describes the SEO architecture used by Sudobility web apps. It covers the template files, runtime components, locale file conventions, and the `@sudobility/seo_lib` library.

## Overview

SEO is implemented in two layers:

```
@sudobility/seo_lib (shared library)
  - SEO component (react-helmet-async wrapper)
  - AIMeta component (AI crawler meta tags)
  - SemanticHTML components (accessible landmarks)
  - advancedSEO utilities (Schema.org generators)
  - aiTrainingMetadata utilities (concept graphs, Q&A)
  - seo-headings utilities (semantic heading validation)

App-level (per project)
  - index_template.html (static meta, structured data)
  - sitemap.template.xml / generate-sitemap.ts
  - llms.template.txt (AI assistant context)
  - manifest.template.json (PWA metadata)
  - robots.txt (crawler directives)
  - SEOHead component (runtime per-route meta)
  - Locale files (per-page SEO strings)
  - src/config/seo.ts (global schemas, breadcrumbs)
  - src/config/pageSEO.ts (static page metadata)
```

## Template Files

Template files use `{{VITE_APP_DOMAIN}}`, `{{VITE_APP_NAME}}`, `{{VITE_TWITTER_HANDLE}}`, `{{VITE_APP_HANDLE}}`, and `{{TODAY}}` placeholders. The `process:static` build step replaces these with environment variable values. The `generate-sitemap.ts` script also regenerates the sitemap and index.html.

### index_template.html

The root HTML template. Contains static SEO that applies before React hydrates.

**What belongs here:**

- `<title>` and `<meta name="description">` for the homepage (overridden per-route by SEOHead)
- Open Graph and Twitter Card tags for the homepage
- Structured data that applies site-wide (Organization, WebSite, WebApplication, HowTo)
- Security headers, viewport, theme-color
- Favicon and icon links
- PWA manifest link
- DNS prefetch and preconnect for critical domains
- Critical inline CSS for LCP
- Font loading (deferred)

**What does NOT belong here:**

- Fabricated meta tags (`ai-*`, `geo-*`, `ml-*`, self-assigned scores). No search engine or AI crawler recognizes these. They add page weight with zero benefit.
- Fake `aggregateRating` or `review` data. Google penalizes fabricated review markup with manual actions.
- Invalid Schema.org types. Only use types from https://schema.org/docs/full.html.
- Hidden `display:none` content blocks. Google may classify hidden text as cloaking.
- Duplicate structured data schemas (e.g., two `SoftwareApplication` blocks for the same product).
- Page-specific meta tags. These go in SEOHead at the route level.
- Hardcoded dates. Use `{{TODAY}}` for `dateModified` fields.

**Title format:**

```
{{VITE_APP_NAME}} - Primary Value Proposition | Secondary Qualifier
```

Keep under 60 characters. Use `{{VITE_APP_NAME}}` (brand name), not `{{VITE_APP_DOMAIN}}`.

**Structured data guidelines:**

Only include schemas that are accurate and verifiable:

| Schema            | Use | Notes                                                                                                                                   |
| ----------------- | --- | --------------------------------------------------------------------------------------------------------------------------------------- |
| `WebApplication`  | Yes | Core product definition. Include `offers` with real pricing, `featureList`, `dateModified: {{TODAY}}`. No fabricated `aggregateRating`. |
| `Organization`    | Yes | Company identity. Only include `legalName` if legally registered. Only include `contactPoint` with real email addresses.                |
| `WebSite`         | Yes | Site definition with `SearchAction` if search exists. Only use valid Schema.org action types.                                           |
| `HowTo`           | Yes | Getting started guide. Must reflect actual product steps.                                                                               |
| `TechArticle`     | No  | Homepage is not an article. Use on actual article pages via SEOHead.                                                                    |
| `Course`          | No  | Product is not a course. Misleading schema type.                                                                                        |
| `AggregateRating` | No  | Only if sourced from real user reviews with a verifiable review system.                                                                 |

### sitemap.template.xml / generate-sitemap.ts

The sitemap is generated by `scripts/generate-sitemap.ts` which reads routes and languages to produce a multilingual sitemap with hreflang tags.

**Route configuration:**

```typescript
const ROUTES = [
  { path: '/', priority: 1.0, changefreq: 'daily' },
  { path: '/connect', priority: 0.9, changefreq: 'weekly' },
  // ...
];
```

**Rules:**

- Every public route must be in the ROUTES array
- Protected routes (requiring authentication) must NOT be in the sitemap
- Internal/debug routes must NOT be in the sitemap
- Languages are imported from the app's `SUPPORTED_LANGUAGES` constant to stay in sync with hreflang tags in SEOHead
- Run `bun run generate:sitemap` after adding routes

**Multilingual hreflang in the sitemap:**

Every `<url>` entry must include `<xhtml:link rel="alternate">` tags for ALL supported languages plus `x-default`. This tells search engines which language variant to show for each locale. The sitemap XML namespace must include `xmlns:xhtml="http://www.w3.org/1999/xhtml"`.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9"
        xmlns:xhtml="http://www.w3.org/1999/xhtml">
  <url>
    <loc>https://example.com/en/about</loc>
    <changefreq>monthly</changefreq>
    <priority>0.7</priority>
    <lastmod>2026-04-11</lastmod>
    <!-- One xhtml:link for EVERY supported language -->
    <xhtml:link rel="alternate" hreflang="en" href="https://example.com/en/about"/>
    <xhtml:link rel="alternate" hreflang="ar" href="https://example.com/ar/about"/>
    <xhtml:link rel="alternate" hreflang="de" href="https://example.com/de/about"/>
    <xhtml:link rel="alternate" hreflang="es" href="https://example.com/es/about"/>
    <!-- ... one for each language ... -->
    <xhtml:link rel="alternate" hreflang="zh-hant" href="https://example.com/zh-hant/about"/>
    <!-- x-default points to the English variant -->
    <xhtml:link rel="alternate" hreflang="x-default" href="https://example.com/en/about"/>
  </url>
</urlset>
```

The `generate-sitemap.ts` script produces this automatically for every route x language combination. Each language variant of a route gets its own `<url>` entry, and every entry contains the full set of alternates. This means for N routes and L languages, the sitemap contains N x L `<url>` entries, each with L+1 `<xhtml:link>` tags (all languages + x-default).

**Key requirements:**

- Hreflang links must be **bidirectional** — if `/en/about` points to `/de/about`, then `/de/about` must also point to `/en/about`. The script handles this by including all alternates in every entry.
- `x-default` must always be present, pointing to the English variant (the default fallback).
- The language codes must match exactly between the sitemap, the `<html lang>` attribute, and the hreflang tags in SEOHead. All three pull from the same `SUPPORTED_LANGUAGES` constant.
- After running `generate-sitemap.ts`, copy the output to `sitemap.template.xml` so the template stays in sync.

**Priority guidelines:**

| Page type                                            | Priority | Changefreq |
| ---------------------------------------------------- | -------- | ---------- |
| Homepage                                             | 1.0      | daily      |
| Core features (connect, docs, governance)            | 0.8-0.9  | weekly     |
| Content pages (about, users, developers, comparison) | 0.7-0.8  | monthly    |
| Sub-pages (governance/_, users/_, developers/\*)     | 0.6-0.7  | monthly    |
| Legal pages (terms, privacy, cookies)                | 0.4      | yearly     |

### llms.template.txt

Plain text file for AI assistants (ChatGPT, Claude, Perplexity). Placed at `/llms.txt` following the emerging `llms.txt` convention.

**Keep it:**

- Factually accurate (correct framework versions, language counts, pricing)
- Updated when features change
- Listing all public pages with descriptions
- Focused on what AI assistants need to help users

### manifest.template.json

PWA Web App Manifest. Contains app name, description, icons, shortcuts, and screenshots.

**`name`** uses `{{VITE_APP_DOMAIN}}` (full name for install prompts).
**`short_name`** uses `{{VITE_APP_NAME}}` (brand for home screen).

### robots.txt

Not a template file — committed directly since it references the production domain.

**Rules:**

- Allow all legitimate crawlers
- Disallow protected routes (`/*/mail`, `/*/admin`, `/*/settings`, `/api/`)
- Set appropriate crawl delays (search engines: 2-3s, AI crawlers: 5s, social: 1s)
- Point to sitemap URL

## Runtime SEO Components

### SEOHead (app-level)

The primary per-route SEO component. Wraps `react-helmet-async` with app-specific logic.

```tsx
<SEOHead
  emailDomain={emailDomain}
  appName={appName}
  title={t('seo.title', { emailDomain })}
  description={t('seo.description', { emailDomain })}
  keywords={t('seo.keywords', { returnObjects: true })}
  ogType="website"
/>
```

**Features:**

- Auto-generates hreflang tags for all supported languages
- Self-referential canonical URL from current pathname (strips query params)
- Auto `noindex` on dev/staging/preview domains
- Supports structured data: FAQ, breadcrumb, HowTo, article schemas
- Governance-specific meta tags via `isGovernancePage` prop
- AI meta tags via `createAIMetaTags()` from `@sudobility/seo_lib`

**Protected pages** must set `noIndex={true}`:

```tsx
<SEOHead
  noIndex={true}
  title={t('seo.title')}
  // ...
/>
```

### SEO component (@sudobility/seo_lib)

The library's core component. Lower-level than SEOHead.

```tsx
import { SEO } from '@sudobility/seo_lib';

<SEO
  title="Page Title"
  description="Page description"
  config={{ appName: 'MyApp', baseUrl: 'https://example.com' }}
  structuredData={[schema1, schema2]}
/>;
```

### AIMeta component (@sudobility/seo_lib)

Renders custom meta tags for AI crawlers.

```tsx
import { AIMeta } from '@sudobility/seo_lib';

<AIMeta
  contentType="tutorial"
  summary="How to connect a wallet"
  complexity="beginner"
  blockchainNetworks={['Ethereum', 'Solana']}
/>;
```

### SemanticHTML components (@sudobility/seo_lib)

Accessible landmark components with proper ARIA roles:

```tsx
import { Main, Nav, Article, Section, H1, H2 } from '@sudobility/seo_lib';

<Main>
  <Article>
    <H1>Page Title</H1>
    <Section>
      <H2>Section Title</H2>
      <p>Content</p>
    </Section>
  </Article>
</Main>;
```

Available: `Main`, `Header`, `Footer`, `Nav`, `Article`, `Section`, `Aside`, `Figure`, `H1`-`H4`, `OrderedList`, `UnorderedList`, `SemanticForm`, `SemanticInput`, `SemanticButton`, `SemanticLink`, `SkipLink`, `ScreenReaderOnly`, `SearchRegion`, `SemanticLoading`.

## Locale File SEO Convention

Each page's SEO strings live in its locale JSON file under the `seo` object or as flat `seoTitle`/`seoDescription`/`seoKeywords` keys.

### Nested pattern (preferred)

```json
{
  "seo": {
    "title": "Page Title | {{emailDomain}}",
    "description": "150-160 char description with primary keyword.",
    "keywords": ["primary keyword", "secondary keyword", "tertiary keyword"]
  }
}
```

### Flat pattern (legacy)

```json
{
  "seoTitle": "Page Title | {{emailDomain}}",
  "seoDescription": "150-160 char description.",
  "seoKeywords": ["keyword1", "keyword2"]
}
```

### Title guidelines

| Rule                            | Example                                                                            |
| ------------------------------- | ---------------------------------------------------------------------------------- |
| Under 60 characters             | `Web3 Email Documentation & API \| {{emailDomain}}`                                |
| Front-load primary keyword      | `DAO Governance Email Tools \| {{emailDomain}}` NOT `{{emailDomain}} \| DAO Tools` |
| Use `{{emailDomain}}` for brand | Replaced at runtime with the actual domain                                         |
| One primary topic per page      | Don't stuff multiple topics into the title                                         |
| No unverifiable claims          | No "Boost by 85%", "Revolutionary", "#1 Platform"                                  |

### Description guidelines

- 150-160 characters
- Include the primary keyword naturally
- Action-oriented: tell users what they can do
- Include `{{emailDomain}}` once for brand recognition

### Keyword guidelines

- 5-8 keywords per page (focused, not exhaustive)
- Each keyword should primarily belong to ONE page (avoid cannibalization)
- Include the primary keyword and close variants
- No keyword should appear in more than 2 pages' keyword lists

### Keyword ownership map

Each public page should own a distinct primary topic. Assign keywords so they don't compete:

| Page          | Primary keyword focus                           | Avoid overlap with                  |
| ------------- | ----------------------------------------------- | ----------------------------------- |
| Landing       | "Web3 email platform", "free Web3 email"        | Everything else                     |
| Connect       | "connect wallet email", "Web3 login"            | Landing (wallet), Tutorials (setup) |
| About         | "about [brand]", "Web3 email company"           | Landing (platform)                  |
| Documentation | "Web3 email docs", "email API"                  | Tutorials (guides)                  |
| Governance    | "DAO governance email", "voting notifications"  | Landing (DAO mention)               |
| Points        | "Web3 email rewards", "points dashboard"        | Token (tokenomics), How to Earn     |
| Token         | "Web3 email token", "tokenomics"                | Points (rewards)                    |
| Comparison    | "[brand] vs EtherMail", "Web3 email comparison" | Landing (features)                  |
| Tutorials     | "Web3 email tutorials", "getting started"       | Docs (reference), Connect (wallet)  |
| How to Earn   | "how to earn crypto email points"               | Points (dashboard)                  |

### All strings must be localized

Every user-visible string on every page — headings, body text, button labels, CTAs, alt text, structured data fields, error messages — must go through the i18n system via `t()` calls. Never hardcode English strings in components. This includes:

- SEO fields: `seo.title`, `seo.description`, `seo.keywords`
- Social sharing: `share.title`, `share.description`, `share.hashtags`
- Structured data: `structuredData.*` fields
- Page content: headings, paragraphs, labels, placeholders, tooltips
- AI metadata: `aiMeta.aiSummary`, `aiMeta.useCase`, etc.

Search engines index content in the language declared by the page's `lang` attribute and hreflang tags. If content is English on a page declared as Japanese, Google may ignore the page or flag it as a language mismatch, hurting rankings for all language variants.

### Localization workflow for SEO changes

When optimizing SEO — whether adding new strings, modifying existing strings, or restructuring page content:

1. **Work on English locale files only.** All changes go in `public/locales/en/<page>.json`.
2. **For any modified or new key**, delete that same key from all 15 non-English locale files (`ar`, `de`, `es`, `fr`, `it`, `ja`, `ko`, `pt`, `ru`, `sv`, `th`, `uk`, `vi`, `zh`, `zh-hant`). This ensures stale translations don't persist.
3. **Run the localization script** to regenerate translations for all languages. Until the script runs, missing keys fall back to English via i18next, which is acceptable temporarily.
4. **Never edit non-English locale files by hand** for SEO content. The localization script is the source of truth for translations.

This applies to all string changes, not just SEO fields. If you rewrite a heading or CTA for SEO purposes, the same workflow applies: update English, delete the key from other locales, run the script.

### Adding SEO for a new page

1. Add `seo.title`, `seo.description`, `seo.keywords` to the English locale file only
2. Choose a primary keyword focus that doesn't overlap with existing pages
3. Add all page content strings to the English locale file (never hardcode in components)
4. Add the route to `scripts/generate-sitemap.ts`
5. Use SEOHead in the page component with `t('seo.title')` etc.
6. Add structured data if appropriate (FAQ, HowTo, Article)
7. Run the localization script to generate all non-English translations

### Modifying existing SEO strings

When changing `seo.title`, `seo.description`, `seo.keywords`, or any other string in the English locale:

1. Update only the English file
2. Remove the same keys from all 15 non-English locale files
3. Run the localization script to regenerate translations
4. Verify no keyword cannibalization with the keyword ownership map

## @sudobility/seo_lib Utilities

### advancedSEO.ts

Schema.org structured data generators:

```typescript
import {
  createWeb3ProductSchema,
  createTechnicalArticleSchema,
  createEnhancedFAQSchema,
  createAIOptimizedSchema,
  createEnhancedOpenGraph,
  createEnhancedTwitterCard,
  createAIMetaTags,
  generateAdvancedSEO,
} from '@sudobility/seo_lib';

// Generate all SEO data at once
const seo = generateAdvancedSEO({
  title: 'Page Title',
  description: 'Page description',
  keywords: ['keyword1', 'keyword2'],
  branding: { appName: 'MyApp', baseUrl: 'https://example.com' },
  category: 'Technology',
  audience: 'developers',
  complexity: 'intermediate',
  contentType: 'guide',
});
// Returns: { structuredData, openGraph, twitterCard, aiMetaTags, jsonLD }
```

Pre-built configs available via `pageSEOConfigs.homepage`, `pageSEOConfigs.documentation`, `pageSEOConfigs.earnPoints`.

### aiTrainingMetadata.ts

Generates AI-readable metadata and concept graphs:

```typescript
import {
  generateAIMetadata, // Schema.org LearningResource
  generateConceptGraph, // { nodes, edges } graph
  generateTrainingExamples, // Labeled examples for a concept
  generateQAPairs, // Q&A pairs from concepts
  WEB3_CONCEPTS, // Pre-built Web3 concept definitions
  AI_TRAINING_CONFIGS, // Pre-built training configs
} from '@sudobility/seo_lib';
```

### seo-headings.ts

Semantic heading utilities with validation:

```typescript
import {
  createSemanticHeading,
  validateHeadingStructure,
  WEB3_HEADING_PATTERNS,
} from '@sudobility/seo_lib';

// Validate heading hierarchy
const result = validateHeadingStructure([
  { level: 1, text: 'Page Title' },
  { level: 2, text: 'Section' },
  { level: 4, text: 'Subsection' }, // Error: skipped H3
]);
// result.errors: ["Heading level skipped from 2 to 4"]
```

**Heading rules:**

- Exactly one H1 per page
- No skipping levels (H1 → H3 without H2 is invalid)
- Headings should be descriptive (3+ chars)
- Keep under 70 chars for SEO

## Structured Data Guidelines

### Valid schemas to use

| Schema                | When to use                       | Key fields                                           |
| --------------------- | --------------------------------- | ---------------------------------------------------- |
| `WebApplication`      | Site-wide (index_template.html)   | name, description, featureList, offers, dateModified |
| `Organization`        | Site-wide (index_template.html)   | name, url, logo, contactPoint, sameAs                |
| `WebSite`             | Site-wide (index_template.html)   | name, url, potentialAction (SearchAction)            |
| `HowTo`               | Getting started / tutorial pages  | name, step[], totalTime, estimatedCost               |
| `FAQPage`             | Pages with FAQ sections           | mainEntity[].name, mainEntity[].acceptedAnswer       |
| `TechArticle`         | Documentation / technical content | headline, author, datePublished, teaches             |
| `BreadcrumbList`      | Pages with breadcrumb navigation  | itemListElement[].name, .item                        |
| `SoftwareApplication` | Product pages                     | Same as WebApplication with platform-specific fields |
| `Product`             | Pricing / subscription pages      | name, offers, description                            |

### Schemas to avoid

| Schema            | Why                                                     |
| ----------------- | ------------------------------------------------------- |
| `AggregateRating` | Only with real, verifiable reviews from a review system |
| `Review`          | Only with real user reviews, never fabricated           |
| `Course`          | Only for actual educational courses with curriculum     |
| `NewsArticle`     | Only for time-sensitive news content                    |
| Any invalid type  | Always verify at https://schema.org/docs/full.html      |

### Validation

Test structured data at:

- https://validator.schema.org (Schema.org validator)
- https://search.google.com/test/rich-results (Google Rich Results Test)

## Checklist for New Pages

- [ ] Page has SEOHead with localized `title`, `description`, `keywords`
- [ ] Title is under 60 characters and front-loads the primary keyword
- [ ] Description is 150-160 characters and action-oriented
- [ ] Keywords are 5-8 focused terms with no cannibalization
- [ ] Protected pages have `noIndex={true}`
- [ ] Route is added to `scripts/generate-sitemap.ts` (if public)
- [ ] English locale file has `seo` object
- [ ] No hardcoded brand names — uses `{{emailDomain}}` or `{{appName}}` template variables
- [ ] Structured data uses only valid Schema.org types with real data
- [ ] No fabricated ratings, reviews, or unverifiable claims
- [ ] No hidden text content (`display:none` with text)
- [ ] Heading hierarchy is valid (single H1, no skipped levels)

## Anti-Patterns

**Do not:**

- Invent meta tag names. Stick to recognized standards (Open Graph, Twitter Cards, standard HTML meta).
- Add self-assigned quality scores (`content-score: 98`, `ai-readability: high`). No crawler reads these.
- Hide text for crawlers using `display:none`. This is classified as cloaking.
- Duplicate Schema.org types for the same entity (two `SoftwareApplication` schemas with different versions).
- Use Schema.org types that don't match the content (a homepage is not a `Course` or `TechArticle`).
- Stuff keywords across many pages. Each page should own one topic.
- Hardcode dates in structured data. Use `{{TODAY}}` template variable or omit.
- Use fake verification codes as placeholders. Only add `google-site-verification` etc. with real values.
- Add `<h1>` tags in hidden content. Crawlers detect duplicate/hidden headings.
